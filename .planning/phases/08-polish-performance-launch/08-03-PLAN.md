# Plan 08-03: Monitoring, Analytics & Observability

## Goal
Vercel Analytics active on frontend; server logs use structured `[INFO]`/`[WARN]`/`[ERROR]`/`[FATAL]` prefixes; game lifecycle events logged for debugging; health endpoint validates approvedQuestions >= 200.

## Prerequisites
None — this plan can start in parallel with Plans 08-01, 08-02, 08-04. No shared files.

**Decision: No Sentry.** Railway log drain + `process.on` handlers (Plan 08-01) are sufficient for MVP. Sentry is deferred.

## Threat Model
<threat_model>
| Threat | STRIDE | Mitigation |
|--------|--------|------------|
| Analytics script blocked by ad-blocker | Denial of Service | Accept — Vercel Analytics degrades gracefully, no runtime error |
| Sensitive data in structured logs (player names, IPs) | Information Disclosure | Log only room codes, socket IDs, and event names — never player names, IPs, or tokens |
| Health endpoint returns 200 when questions below launch threshold | Tampering | Add `approvedQuestions >= 200` check; return 503 if below threshold |
</threat_model>

## Tasks

### Task 1: Install and add Vercel Analytics to frontend
**File(s):** `apps/web/package.json`, `apps/web/app/layout.tsx`
**What:**
- Run `cd apps/web && npm install @vercel/analytics`.
- In `apps/web/app/layout.tsx`:
  - Add `import { Analytics } from '@vercel/analytics/react'` at top.
  - Inside the `<body>` tag, after `{children}`, add `<Analytics />`.
  - The component auto-detects Vercel environment; no env vars needed. In non-Vercel environments it renders nothing (no runtime error).
**Acceptance:** `cd apps/web && npx next build` succeeds. `grep -c "Analytics" apps/web/app/layout.tsx` returns >= 2 (import + usage).

### Task 2: Standardize server log prefixes
**File(s):** `apps/server/src/index.ts`, `apps/server/src/socket/index.ts`
**What:**
- In `index.ts`:
  - Change the `console.warn('[WARN]...')` line to keep `[WARN]` prefix (already correct).
  - Change `console.error('[Error]', err.message)` in the global error handler to `console.error('[ERROR] Express:', err.message)`.
  - Change `console.log('[Server] Listening...')` to `console.log('[INFO] Server listening on port ${PORT}')`.
- In `socket/index.ts`:
  - Change `console.log('[Socket.io] Client connected...')` to `console.log('[INFO] Socket connected: ${socket.id}')`.
  - Change `console.log('[Socket.io] Client disconnected...')` to `console.log('[INFO] Socket disconnected: ${socket.id} — ${reason}')`.
- Do NOT change any existing `[Room]` or `[Game]` prefixed logs in room.ts or game.ts — those are addressed in Task 3 and Task 4.
**Acceptance:** `cd apps/server && npx tsc --noEmit` passes. `grep -c "\\[INFO\\]" apps/server/src/index.ts` returns >= 1. `grep -c "\\[ERROR\\]" apps/server/src/index.ts` returns >= 1.

### Task 3: Add structured log prefixes to room.ts
**File(s):** `apps/server/src/socket/room.ts`
**What:**
- Replace all `console.log('[Room]...')` with `console.log('[INFO] Room:...')`.
- Replace all `console.error(...)` with `console.error('[ERROR] Room:...')` (keep the error content).
- Add lifecycle logs where missing:
  - After successful `room:join`: `console.log('[INFO] Room: ${name} joined ${roomCode} (${room.players.length}/8 players)')` (log player count for capacity monitoring).
  - After `room:start`: `console.log('[INFO] Room: game started in ${roomCode}')`.
  - After `room:end`: `console.log('[INFO] Room: game ended in ${roomCode}')`.
- Do NOT log player names in any new log lines — use socket.id or room code only. The existing `[Room] ${name} joined` log is acceptable since it is the display name (not PII).
**Acceptance:** `cd apps/server && npx tsc --noEmit` passes. `grep -c "\\[INFO\\] Room:" apps/server/src/socket/room.ts` returns >= 4.

### Task 4: Add structured log prefixes to game.ts
**File(s):** `apps/server/src/socket/game.ts`
**What:**
- Replace all `console.log('[Game]...')` with `console.log('[INFO] Game:...')`.
- Replace all `console.error('[Game]...')` with `console.error('[ERROR] Game:...')`.
- Replace all `console.warn('[Game]...')` with `console.warn('[WARN] Game:...')`.
- Replace `console.error('[Analytics]...')` with `console.error('[WARN] Analytics:...')` (downgrade — analytics failure is non-critical).
- These are find-and-replace operations on existing log lines. Do NOT add new log lines.
**Acceptance:** `cd apps/server && npx tsc --noEmit` passes. `grep -c "\\[INFO\\] Game:" apps/server/src/socket/game.ts` returns >= 5. `grep -c "console.log\('\[Game\]" apps/server/src/socket/game.ts` returns 0 (old format removed).

### Task 5: Update health endpoint threshold to >= 200
**File(s):** `apps/server/src/routes/health.ts`
**What:**
- After `checks.approvedQuestions = await prisma.question.count(...)`, add:
  ```
  if (typeof checks.approvedQuestions === 'number' && checks.approvedQuestions < 200) {
    checks.questionThreshold = 'below_minimum'
  }
  ```
- Update the `allOk` check to also fail if `checks.questionThreshold === 'below_minimum'`:
  ```
  const allOk = ['postgres', 'redis'].every((k) => checks[k] !== 'error') && checks.questionThreshold !== 'below_minimum'
  ```
- This makes `/health` return 503 if approved questions < 200, which the E2E tests in Plan 08-05 will verify.
**Acceptance:** `cd apps/server && npx tsc --noEmit` passes. `grep -c "200" apps/server/src/routes/health.ts` returns >= 1 (the threshold). `grep -c "questionThreshold" apps/server/src/routes/health.ts` returns >= 2.

### Task 6: Update health endpoint test for new threshold
**File(s):** `apps/server/src/routes/health.test.ts`
**What:**
- Read the existing test file to understand the mock structure.
- Add or update a test: "returns 503 when approvedQuestions < 200". Mock `prisma.question.count` to return 50. Assert response status is 503 and body includes `questionThreshold: 'below_minimum'`.
- Add a test: "returns 200 when approvedQuestions >= 200". Mock `prisma.question.count` to return 201. Assert response status is 200.
**Acceptance:** `cd apps/server && npm run test -- --run src/routes/health.test.ts` passes with the threshold tests present.
