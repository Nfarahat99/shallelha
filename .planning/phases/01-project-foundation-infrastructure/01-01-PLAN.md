  <action>
Task 1 created all local config files. Railway and Vercel accounts and projects must now be created in the cloud. No CLI automation is possible without prior account authentication.

Complete these steps in order:

**Railway setup (backend + database + Redis):**
1. Go to https://railway.app — create an account or log in
2. Click "New Project" → "Empty Project" — name it "shallelha"
3. Inside the project, click "+ New" → "Database" → "Add PostgreSQL" — Railway provisions it automatically
4. Click "+ New" again → "Database" → "Add Redis" — Railway provisions it automatically
5. Click "+ New" → "GitHub Repo" → connect this repository → select the shallelha repo
6. Railway will detect railway.json and use the Dockerfile builder
7. Go to the Node.js service → "Variables" tab → add these variables:
   - DATABASE_URL = ${{Postgres.DATABASE_URL}} (Railway reference variable — exactly as written)
   - REDIS_URL = ${{Redis.REDIS_PRIVATE_URL}}?family=0 (append ?family=0 manually)
   - FRONTEND_URL = leave blank for now (set after Vercel deploy)
   - NODE_ENV = production
   - PORT = 4000
8. Note the Railway public URL from the service's "Settings" tab (e.g. https://shallelha-server-xxxx.up.railway.app)

**Vercel setup (frontend):**
1. Go to https://vercel.com — create an account or log in
2. Click "Add New Project" → import the shallelha GitHub repository
3. Leave root directory blank — Vercel will use vercel.json
4. Under "Environment Variables", add:
   - NEXT_PUBLIC_BACKEND_URL = the Railway public URL from step 8 above
5. Click "Deploy"
6. Note the Vercel URL (e.g. https://shallelha.vercel.app)

**Complete the Railway FRONTEND_URL:**
7. Go back to Railway → Node.js service → Variables → set FRONTEND_URL = the Vercel URL

**IMPORTANT — Railway project structure:** All three services (Node.js, PostgreSQL, Redis) MUST be in the same Railway project, not separate projects. Same project = private networking. Separate projects = no private network, added latency and egress costs.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer machine → git remote | Secrets must never enter the repo; .gitignore enforces this |
| Railway service → PostgreSQL/Redis | Private networking via *.railway.internal; no public exposure needed |
| Vercel frontend → Railway backend | Cross-origin; CORS origin locked to FRONTEND_URL env var |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | .env files | mitigate | .gitignore excludes all .env variants; .env.example has no real secrets — only placeholder values |
| T-01-02 | Information Disclosure | Railway Variables tab | accept | Railway Variables are access-controlled by Railway account; no additional mitigation needed at this phase |
| T-01-03 | Elevation of Privilege | CORS wildcard | mitigate | FRONTEND_URL env var used as explicit CORS origin in Plan 02; wildcard origin never used |
| T-01-04 | Information Disclosure | Stack traces in HTTP 500 | mitigate | NODE_ENV=production set in Railway; error handler in Plan 02 returns generic message in production |
</threat_model>
