**Step 3: Create `apps/web/auth.ts` — NextAuth v5 configuration.**

Per locked decision D-01 (Google OAuth + Resend magic link) and D-06 (JWT sessions):

```typescript
import NextAuth from "next-auth"
import Google from "next-auth/providers/google"
import Resend from "next-auth/providers/resend"
import { PrismaAdapter } from "@auth/prisma-adapter"
import { prisma } from "@/lib/prisma"

export const { handlers, auth, signIn, signOut } = NextAuth({
  adapter: PrismaAdapter(prisma),
  providers: [
    Google({
      clientId: process.env.AUTH_GOOGLE_ID,
      clientSecret: process.env.AUTH_GOOGLE_SECRET,
    }),
    Resend({
      apiKey: process.env.AUTH_RESEND_KEY,
      from: "noreply@shallelha.com",
    }),
  ],
  session: { strategy: "jwt" },
  pages: {
    signIn: "/auth/signin",
    error: "/auth/error",
  },
  callbacks: {
    jwt({ token, user }) {
      if (user) {
        token.id = user.id
      }
      return token
    },
    session({ session, token }) {
      session.user.id = token.id as string
      return session
    },
  },
})
```

Per D-06: Use JWT session strategy. The Prisma adapter is STILL required for OAuth account linking (Google) and magic link token storage (VerificationToken table). JWT means session reads don't hit the DB — only auth operations (sign-in, link account) do.

Per RESEARCH.md: env vars use `AUTH_` prefix in v5 (not `NEXTAUTH_`). `AUTH_SECRET` is auto-detected by NextAuth v5.

**Step 5: Create `apps/web/middleware.ts` — route protection for /host paths.**

Per locked decision D-03, `/host` and `/host/[roomCode]` require authentication. `/join` does NOT.

```typescript
import { auth } from "@/auth"
import { NextResponse } from "next/server"

export default auth((req) => {
  const { pathname } = req.nextUrl

  if (pathname.startsWith("/host") && !req.auth) {
    const signInUrl = new URL("/auth/signin", req.nextUrl.origin)
    signInUrl.searchParams.set("callbackUrl", pathname)
    return NextResponse.redirect(signInUrl)
  }

  return NextResponse.next()
})

export const config = {
  matcher: ["/host", "/host/:path*"],
}
```

Per RESEARCH.md Pattern 3: middleware is a UX convenience, not the security boundary. Server Components should also verify `auth()` before sensitive operations.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> NextAuth API route | Untrusted sign-in requests (OAuth callback, magic link verification) |
| NextAuth -> Google OAuth | External OAuth provider — token exchange |
| NextAuth -> Resend API | External email provider — magic link delivery |
| NextAuth -> PostgreSQL | Adapter writes (account linking, token storage) |
| Browser -> /host routes | Unauthenticated users must be redirected |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | NextAuth sign-in | mitigate | NextAuth v5 handles CSRF tokens, OAuth state params, and PKCE automatically |
| T-02-02 | Information Disclosure | AUTH_SECRET env var | mitigate | Must be server-only env var; NEVER prefix with NEXT_PUBLIC_; documented in .env.example with warning comment |
| T-02-03 | Elevation of Privilege | /host route access without auth | mitigate | middleware.ts redirects unauthenticated /host requests; Server Components also verify auth() before sensitive operations |
| T-02-04 | Information Disclosure | DATABASE_URL in Vercel (public Railway URL) | accept | Required for Prisma adapter; Railway PostgreSQL still requires password authentication; acceptable for MVP |
| T-02-05 | Tampering | JWT session token | mitigate | AUTH_SECRET signs JWTs; NextAuth uses HS256 by default; token cannot be forged without secret |
| T-02-06 | Information Disclosure | Stack traces in auth errors | mitigate | NextAuth v5 uses custom error pages (/auth/error); no stack traces exposed to client |
</threat_model>
