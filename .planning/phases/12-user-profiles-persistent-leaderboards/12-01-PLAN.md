2. On the PlayerGameResult model:
   a. Change `userId String` to `userId String?` (make FK nullable so anonymous players can be stored)
   b. Change `user User @relation(fields: [userId], references: [id], onDelete: Cascade)` to:
      ```
      user          User?       @relation(fields: [userId], references: [id], onDelete: SetNull)
      ```
      (Change to optional relation with SetNull so deleting a User does not cascade-delete anonymous results)
   c. Add `avatarConfig Json?` after `playerEmoji String?`
   d. Update the @@unique constraint: change `@@unique([gameSessionId, userId])` to:
      ```
      @@index([userId, createdAt])
      @@index([isWinner, createdAt])
      ```
      ALSO keep the unique constraint but make it conditional — remove the @@unique([gameSessionId, userId]) entirely (it cannot be enforced when userId is nullable in PostgreSQL NULLs-are-distinct semantics). Add instead a partial unique index via raw migration SQL (document in a comment, do NOT add a @@unique Prisma attribute).

NOTE: The @@unique([gameSessionId, userId]) removal is intentional — PostgreSQL treats NULL != NULL in unique constraints, so multiple anonymous rows (userId=null) in the same session become possible. This is acceptable for Phase 12 since anonymous players are differentiated by playerName within a session.

